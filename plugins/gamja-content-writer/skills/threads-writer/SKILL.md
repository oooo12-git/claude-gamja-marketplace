# 쓰레드 작성 가이드

## 어투 규칙 (가장 중요)

반말 + 일상 대화체를 사용해야 함.

### 변환 예시

| 변환 전 (X) | 변환 후 (O) |
|------------|------------|
| 엇갈렸습니다 | 엇갈렸거든 |
| 놀랐습니다 | 놀랐어 |
| ~할 수 있습니다 | ~할 수 있어 |
| 확인해보세요 | 확인해봐 |
| 필요합니다 | 필요해 |

### 어투 특징
- 친구에게 말하듯이 작성
- "있잖아", "거든", "딱 좋음" 같은 구어체 사용
- 문장 끝을 딱딱하게 마무리하지 않기

### 피해야 할 표현

| 피하기 (X) | 대신 사용 (O) | 이유 |
|-----------|--------------|------|
| 명사형 종결 (빗나감) | 동사형 종결 (빗나갔어) | 더 자연스러운 대화체 |
| 단정적 명령 (돌아봐) | 질문형 (돌아볼 필요가 있지 않을까?) | 톤 완화, 공감 유도 |
| 3인칭 (벼락치기 장인은) | 1인칭 전환 (내가 벼락치기 장인은) | 독자가 자신의 이야기로 느낌 |

### 이모티콘 활용
- 부정적/자조적 내용 뒤에 "ㅎㅎ" 추가로 톤 완화
- 과도한 사용 금지 (thread당 최대 1개)

### 부정적 내용의 공감 완충 표현

불합격, 실패 등 부정적 내용을 언급할 때 **바로 지적하지 말고 공감 표현을 앞에 추가**

| 피하기 (X) | 대신 사용 (O) |
|-----------|--------------|
| 불합격했다면 벼락치기 장인은 아니었는지... | 25년 3회에 떨어졌다면, **아쉽지만** 벼락치기 장인은 아니었는지... |
| 떨어진 사람들은 공부를 안 한 거야 | 이번에 아쉽게 떨어졌다면, 다음엔... |

**효과**:
- "아쉽지만", "이번엔 아쉽게" 같은 완충 표현으로 독자 방어심리 완화
- 바로 지적하면 반감, 공감 먼저 하면 수용

### 검색 키워드 (SEO)
줄임말만 쓰지 말고 정식 명칭도 함께 사용해야 검색에 노출됨

| 줄임말 | 정식 명칭 (반드시 포함) |
|--------|------------------------|
| 정처기 | 정보처리기사 |
| 빅분기 | 빅데이터분석기사 |
| 컴활 | 컴퓨터활용능력 |

**적용 예시:**
- 킥 문장: "정처기" 사용 OK (짧고 임팩트)
- 본문 어딘가: "정보처리기사" 최소 1회 포함

```
(X) 정처기 시험 일정 알려줄게
(O) 정처기 시험 일정 알려줄게 + 본문에 "정보처리기사" 언급
```

### 자조적 표현
일반적이지 않은 경험이나 관련 없어 보이는 정보를 공유할 때, 괄호 안에 자조적 코멘트를 넣어 친근감과 공감 유도

| 상황 | 예시 |
|------|------|
| 오래 걸린 일 | "12학번 (응 12년 걸림)", "24년 졸업 (ㅎㅎ)" |
| 관련 없는 자격증/경력 | "재무위험관리사 (왜 땄는지 나도 몰라)", "마케팅 경력 (지금은 안 씀ㅎㅎ)" |
| 의외의 정보 | "기계공학 전공 (맞아 비전공자야)" |

**효과**: "??" 같은 불명확한 표현보다 구체적인 자조 코멘트가 더 친근하고 공감 유도

## 킥 문장 유형

### 1. 반전형
기존 상식을 뒤집거나 "하지 마세요" 형태
- "애드센스 세금 정보 강의 사지 마세요"
- "비밀처럼 말하는데 특별한 거 없어"

### 2. 수치 약속형
구체적인 효과를 수치로 제시
- "10x faster"
- "17% 세금 아끼는 법"

### 3. 공감형
일상적 불편함/고민에서 시작
- "가끔 맘에 드는 글 나중에 다시 보고 싶을 때 있잖아"
- "정처기 시험 보고 망한 것 같을 때 있지"

### 4. 정보제공형 (권장)
바로 가치를 제공하며 시작 - 경고/호기심보다 효과적인 경우 많음
- "2026년 1회 정처기 전체 일정 정리해둘게"
- "합격률 데이터 분석해봤어"
- "이번 회차 시험 난이도 정리해줄게"

**정보제공형이 효과적인 경우:**
- 일정, 데이터, 정리 등 실용 정보 콘텐츠
- 타겟이 이미 해당 정보가 필요한 사람들일 때
- 저장/공유 유도가 목적일 때

### 킥 문장 작성 시 주의사항
- **숫자는 1개만**: 여러 숫자 나열 금지 (22,652명 중 5,414명 → 23.9%만)
- **전문 용어 제외**: DAU, CTR 등 일반인에게 생소한 용어는 킥에서 빼고 본문에서 설명

## Thread 간 연결

### 결과 예고형 연결 (권장)
- "그런데 뜯어보니까 이유가 뭔지 알겠더라구" (다음 내용에 대한 기대감)
- "근데 이걸 보고 깨달은 게 있어"

### 피해야 할 연결
- "그래서 분석해봤어" → 결과가 뭔지 궁금증 없음
- 단순 나열식 연결

## 쓰레드 구조

### 기본 구조 (3개 권장)
```
Thread 1: 킥 + 핵심 정보 (바로 가치 제공)
Thread 2: 실전 팁/방법 (구체적 step-by-step)
Thread 3: CTA/홍보 (선택)
```

### 정보제공형 콘텐츠 구조
첫 쓰레드에 핵심 정보 + CTA를 함께 넣을 수 있음
```
Thread 1: 핵심 정보 정리 + 링크 (저장 유도)
Thread 2: 실전 팁
Thread 3: 추가 서비스 홍보
```

**주의**: 4개 이상은 길어짐 → 3개 이내로 간결하게

### 첫 쓰레드 가치 집중 전략 (중요)

킥 문장만 쓰지 말고, **첫 쓰레드에 배경 + 핵심 데이터까지 담아야** 함.
- 스크롤 안 해도 핵심 전달됨
- thread 2를 안 봐도 이미 인사이트 파악 가능
- 후속 쓰레드는 결론/인사이트만 (짧고 임팩트 있게)

**비교 예시:**

| 구조 | 효과 |
|------|------|
| thread 1: 킥만 / thread 2: 배경+인사이트 | ❌ thread 2가 길어짐, 첫 쓰레드만 보고 이탈 |
| thread 1: 킥+배경+spoiler 데이터 / thread 2: 인사이트만 | ✅ 첫 쓰레드에서 가치 전달, thread 2는 짧고 임팩트 |

**실제 적용:**
```
(❌ 비효율적)
Thread 1: 응시자 중 23.9%만 붙은 정처기 실기, 이유 분석해봤어.
Thread 2: 25년 3회 정처기 실기... (배경 설명 + 데이터 + 인사이트 = 길어짐)

(✅ 효율적)
Thread 1: 응시자 중 23.9%만 붙은 정처기 실기, 이유 분석해봤어.
          25년 3회 정처기 실기, 기출 분석해보니... 결과는 23.9%. 완전 빗나갔어.
          근데 일일 방문자 수를 뜯어보니까 이유가 보이더라구.
          <!-- spoiler --> 평소: 800명대 / 시험 전날: 3,000명 <!-- spoiler 끝 -->
Thread 2: 꾸준히 공부한 사람 vs 벼락치기 장인으로 가정하면... (인사이트만 = 짧음)
```

### Spoiler Feature 활용 (적극 권장)
핵심 정보를 숨겨서 펼쳐보기 유도
- 체류 시간 증가 → 알고리즘에 긍정적 신호
- **리스트형 콘텐츠: 각 항목마다 spoiler 적용**

**참고**: spoiler 태그는 쓰레드 플랫폼 지원 여부와 무관하게 MD 파일에서 사용. 작성자가 킥 포인트나 핵심 내용을 표시해두고 확인하기 위한 용도로도 활용됨.

```
📝 필기
- 사전입력: <!-- spoiler --> 1/8 14시 ~ 1/11 <!-- spoiler 끝 -->
- 원서접수: <!-- spoiler --> 1/12 10시 ~ 1/15 <!-- spoiler 끝 -->
```

**결과 예고형 연결 + spoiler 조합 (권장 패턴)**

"이유가 보이더라구" 같은 결과 예고 문장 뒤에 spoiler로 데이터를 숨기면:
1. 궁금증 유발 → 펼쳐보기 유도
2. 같은 쓰레드 안에서 기대감 → 해소 완결

```
근데 정처기 감자 일일 방문자 수를 뜯어보니까 이유가 보이더라구.
<!-- spoiler -->
평소: 800명대
시험 전날: 3,000명
<!-- spoiler 끝 -->
```

**핵심**: 예고 문장("이유가 보이더라구")과 spoiler 데이터를 같은 쓰레드에 배치해야 함. 다음 쓰레드로 넘기면 이탈률 증가.

### 링크 제공 방식
- **직접 URL 권장**: "bio 링크 확인해봐" (X) → 직접 URL 제공 (O)
- 이유: 클릭 전환율 높음, 접근성 향상
```
링크 확인 -> https://example.com/page
```

### CTA 용어: 스하리/반하리
쓰레드에서 팔로우/좋아요/리포스트를 유도할 때 자주 쓰는 표현

| 용어 | 의미 | 풀이 |
|------|------|------|
| 스하리 | 팔로우 + 하트(좋아요) + 리포스트 | **ㅅ**팔로우, **하**트, **리**포스트 |
| 반하리 | 스하리 해주면 반대로 나도 스하리 해줄게 | **반**대로 **하**트 **리**포스트 |

**사용 예시:**
```
🔥 스하리하면 반하리 갈게!
🔥 3일간 스하리하면 반하리!
```

**주의**: 기간 제한 CTA("3일간")는 콘텐츠 유효기간을 제한하므로, 상황에 따라 선택

## 스레드 미지원 마크다운

스레드 플랫폼에서 지원하지 않는 마크다운 문법. 콘텐츠에 사용하면 안 됨.

| 문법 | 예시 | 대안 |
|------|------|------|
| 표 (table) | `\| 연도 \| 합격률 \|` | 줄바꿈 리스트나 화살표 형식으로 풀어쓰기 |
| Bold | `**강조**` | 그냥 텍스트로 쓰거나 이모지로 강조 |

**표 대안 예시:**
```
(X) 표 형식
| 연도 | 필기 | 실기 |
| 2025 | 65.7% | 22.1% |

(O) 화살표 형식
2025: 필기 65.7% → 실기 22.1%
2024: 필기 60.8% → 실기 28.9%

(O) 슬래시 형식
2025: 필기 65.7% / 실기 22.1%
```

**Bold 대안:**
```
(X) **그래도 3개월이면 충분해.**
(O) 그래도 3개월이면 충분해.
```

## 개선 시 체크리스트

사용자 쓰레드 개선할 때 반드시 확인:

- [ ] 어투가 반말/대화체인가?
- [ ] 킥 문장이 3초 안에 관심을 끄는가?
- [ ] 바로 적용 가능한 정보인가?
- [ ] 쓰레드 3개 이내로 간결한가?
- [ ] 리스트형이면 spoiler 적용했는가?
- [ ] 링크는 직접 URL로 제공했는가?
- [ ] 불필요하게 긴 설명은 없는가?
- [ ] 미지원 마크다운(표, Bold) 사용하지 않았는가?

## 콘텐츠 유형별 전략

| 유형 | 특징 | 강점 |
|------|------|------|
| 리스트형 + 이미지 | 정리된 정보 나열 | 좋아요/리포스트 높음 |
| 실용 팁 | 즉시 적용 가능 | 공유 압도적 |
| 전문 정보 | 타겟 좁음 | 해당 관심사에 강함 |

## 좋은 쓰레드 공식

1. **킥 문장**: 3초 안에 "이거 나한테 필요해" 느끼게
2. **가치 밀도**: 짧은 글에 바로 써먹을 수 있는 정보
3. **저장 욕구**: "나중에 또 볼 것 같다" → 공유로 이어짐
4. **완결성**: thread 2-3개로 끝내되, 각각 독립적으로 의미 있게

## 파일명 형식

### 파일명 규칙

**포스팅 완료:**
```
{업로드일자}_{주제}.md
```
- 예: `20260105_jeongcheogi-application.md`, `20260108_who-make-reservation.md`
- 디렉터리에서 날짜별로 정렬되어 포스팅 순서 파악 용이

**미포스팅 (새로 작성한 쓰레드):**
```
not-yet_{주제}.md
```
- 예: `not-yet_pass-rate.md`, `not-yet_new-feature.md`
- 포스팅 후 `{업로드일자}_{주제}.md`로 파일명 변경

### 버전 접미사 규칙
파일에 `_v2`, `_v3` 등 접미사가 있으면 가장 높은 버전이 최종본

| 상황 | 최종본 |
|------|--------|
| `20260104_pass-rate.md`, `20260104_pass-rate_v2.md`, `20260104_pass-rate_v3.md` | `_v3`가 최종본 |
| `20260108_who-make-reservation.md` (접미사 없음) | 해당 파일이 최종본 |

**레퍼런스 분석 시:** 접미사 없는 파일 또는 가장 높은 버전만 참고

## MD 파일 작성법

쓰레드 콘텐츠는 아래 구조로 md 파일에 작성:

```markdown
# [Kick sentence] - thread 1
<!-- 목적: 이 쓰레드의 목적 -->
<!-- 내용 -->
킥 문장 내용
<!-- 내용 끝-->

# [Kick Image]
없음 or ![설명](이미지 경로)

# explanation

## Kick sentence에 대한 근거, 설명

### thread 2
<!-- 목적: 이 쓰레드의 목적 -->
<!-- 내용 -->
내용
<!-- 내용 끝 -->

### thread 3
<!-- 목적: 이 쓰레드의 목적 -->
<!-- 내용 -->
내용
<!-- 내용 끝 -->

# 쓰레드 포스팅 결과
## 조회수
## 리포스트
## 공유하기
## 좋아요
## 댓글

# 예상 쓰레드 포스팅 결과
```

### 작성 규칙
- `<!-- 목적: -->`: 각 쓰레드의 의도 명시 (분석용)
- `<!-- 내용 -->` ~ `<!-- 내용 끝 -->`: 실제 포스팅할 내용
- `# 쓰레드 포스팅 결과`: 포스팅 후 성과 기록용 (중요도 순: 조회수 > 리포스트, 공유 > 좋아요, 댓글)
- `# 예상 쓰레드 포스팅 결과`: 작성 시 예상 결과 기록

### Spoiler Feature 표시
```markdown
<!-- spoiler feature 적용-->
숨길 내용
<!-- spoiler feature 적용 끝 -->
```

자세한 레퍼런스 분석은 [references.md](references.md) 참조.

## 포스팅 성과 분석 기준

### 지표 중요도 순서
1. **조회수** (가장 중요)
2. **리포스트, 공유하기**
3. **좋아요, 댓글**

### 좋은 포스팅 판단 기준
조회수가 높은 포스팅 = 잘 쓰여진 포스팅으로 가정

**주의사항:**
- 앞선 포스팅 결과가 좋으면 다음 포스팅도 조회수가 높은 경향 있음
- 따라서 **앞뒤 조회수 대비 갑자기 폭발한 포스팅**이 진짜 좋은 포스팅

**분석 예시:**
```
포스팅 A: 조회수 1,000
포스팅 B: 조회수 1,200  ← 앞 포스팅 영향
포스팅 C: 조회수 5,000  ← 이 포스팅이 진짜 좋은 것
포스팅 D: 조회수 3,000  ← 앞 포스팅 영향
```

## 예상 포스팅 결과 작성 가이드

쓰레드 작성 후 마지막에 예상 결과를 작성하여 나중에 실제 결과와 비교

### 작성 항목
```markdown
# 예상 쓰레드 포스팅 결과
## 예상 조회수
1,500

## 예상 근거
- 킥 문장 유형: 정보제공형
- 타겟: 정처기 원서접수 준비하는 수험생
- 시의성: 원서접수 당일이라 높은 관심 예상
- 참고: 비슷한 주제 이전 포스팅 조회수 1,200

## 예상 강점
- 실용적인 지역별 시간 정보
- spoiler로 체류시간 증가 기대

## 예상 약점
- 단발성 정보라 저장/공유 욕구 낮을 수 있음
```

### 예상 vs 실제 비교 활용
나중에 실제 결과와 비교하여:
- 예상보다 높으면 → 무엇이 효과적이었는지 분석
- 예상보다 낮으면 → 무엇이 부족했는지 분석
